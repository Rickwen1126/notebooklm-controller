# 2026-04-13 Todo

## Context

- This note captures working observations from using `nbctl` as a real NotebookLM management and learning tool.
- It is not a canonical spec. Stable guidance should be promoted into `specs/` or long-lived docs later.

## Observations

- `register_all_notebooks(async=true)` is the correct default for large accounts. Sync scan is too easy to time out at the client layer even when the daemon is healthy.
- `async` query materially improves notebook inventory work. It turns NotebookLM from a single-notebook Q&A tool into a batch triage and classification workflow.
- Alias cleanup plus `unregister_notebook` is high leverage. Once noisy notebooks and generic aliases are removed, the remaining collection becomes meaningfully searchable and usable.
- `gpt-5-mini` is noticeably better than the previous planner model for notebook-inventory prompts such as:
  - summarize notebook topic
  - identify source type
  - suggest a classification name
- Query-based notebook summaries are useful for triage, but they are not sufficient authority for canonical alias naming. A real misclassification happened when a NotebookLM audio/podcast notebook was briefly renamed as a general intro guide.
- Query execution still has a false-success class: one notebook completed with the literal result `No answer was found in the page...` instead of surfacing a structured failure or retry-needed status.
- Another false-success class exists when the notebook is in a source/upload-oriented page state and the expected chat input is unavailable or disabled. The task can still end as `completed` with a diagnostic string instead of a real notebook answer.

## Working Insight

- `nbctl` + async query + alias governance is already a strong learning workflow:
  - scan all notebooks into local management
  - use async query to inventory and classify notebooks
  - rename aliases so the category is obvious from the alias itself
  - remove mixed or low-value notebooks from local management
  - keep a smaller, higher-signal notebook set for daily work

## Suggested Next Steps

- Continue prefix-based alias normalization for remaining technical notebooks, especially tooling groups.
- Consider a dedicated notebook-inventory / classify tool instead of relying on general `exec` prompts for every triage step.
- Keep planner fallback telemetry visible so planner misses can be reviewed and converted into better planner rules or dedicated operations.
- Treat “single canonical intro notebook” as a first-class curation pattern for domains where multiple overlapping notebooks exist.
- Add a safer alias-curation workflow:
  - async query may propose a topic and category
  - canonical alias decisions must cross-check notebook title and, when needed, source inventory
  - if query summary conflicts with title-level evidence, mark the notebook as review-needed instead of auto-renaming
- Improve query result validation:
  - if the answer selector returns no match, do not silently mark the task as a useful completed query result
  - distinguish between true answer content, empty answer, and readback failure
  - expose a structured `readback_failed` / `retry_recommended` signal for inventory workflows
- Improve pre-query readiness checks:
  - detect source-panel / disabled-query-box state before submitting inventory queries
  - surface a structured `notebook_not_query_ready` outcome instead of returning a plain diagnostic string as a completed result
